Overview

Self-hosted live web display for a Viennese school's substitution plan (Supplierplan), built as a replacement for the cluttered WebUntis monitor view. It renders a dark, portrait-oriented, large-screen layout that runs fullscreen on a hallway monitor.

A Python script pulls data from the WebUntis JSON-RPC API on a schedule, applies school-specific filtering and absence logic, and writes a static index.html. A minimal static web server serves it; the browser auto-refreshes. An optional train widget shows live departures from the nearby railway station via the ÖBB HAFAS API.

This document is the technical companion to the README. The README is the comprehensive, self-contained reference (overview, features, setup, full configuration table, architecture). This page goes deeper on the internals the README intentionally keeps brief: the substitution-processing heuristics, the absence logic, the render pipeline, the browser-side layout engine, and the domain vocabulary.

Source of truth. The authoritative, always-current description of behaviour lives in CLAUDE.md at the repository root and in the code itself (scripts/fetch_untis.py). This page is a readable rendering of that knowledge; when in doubt, the code wins.

Glossary

The domain is Austrian school scheduling. Several terms are German and have no clean one-word English equivalent — they are kept verbatim throughout the code and this documentation.

Supplierplan (de)
The substitution / cover plan: which teacher covers which class when the regular teacher is absent. The product's central data set, fetched via WebUntis getSubstitutions.
Supplieren (de)
To substitute / cover another teacher's lesson. A Supplierung is a single cover assignment.
Vertretung (de)
Substitution — a lesson taught by a replacement teacher. Rendered with an arrow old → new (the original teacher struck through).
Entfall (de)
A cancelled lesson (it simply does not take place). Collected into a dedicated “Entfallende Stunden” (cancelled lessons) section at the end of each day.
Aufsicht / Pausenaufsicht (de)
Break supervision — a teacher supervising a corridor/yard during a break. Represented in the API as lesson type bs (lstype="bs") and rendered distinctly.
Raumwechsel (de)
Room change. Shown as oldRoom → newRoom (old room struck through). Driven by the orgid on a room (ro[]) entry.
Klasse (de)
A class group (cohort of pupils), e.g. 4A. Deduplicated per row.
Stunde (de)
A lesson period (0–10), mapped from start/end times via the WebUntis timegrid.
Nächster Schultag (de)
“Next school day” — the upcoming day with lessons, skipping weekends and holidays. Shown alongside or instead of today's plan.
Pseudo teacher
A non-person placeholder in the school's WebUntis master data, e.g. Z Entfall (“cancellation marker”) or Mr. X (“filler”). Treated like the empty marker --- via the SKIP_TEACHERS setting.
--- marker
WebUntis' convention for “no teacher”. Used by the processing logic to infer that a named teacher is absent (see Substitution processing).
Compact mode
A width-triggered rendering mode (narrow columns): art badges become single round letters, “Aufsicht” is abbreviated to “Aufs.”, room column widens.

Architecture

Data flow

┌──────────────────────────────────────────────────────────────┐
│ Host (Proxmox LXC  ·  or Windows tray app)                   │
│                                                              │
│  Schedule */5 min → scripts/fetch_untis.py                   │
│    ├─→ WebUntis JSON-RPC  (login as service account)         │
│    ├─→ weekly/data REST   (per-teacher / class absence)      │
│    ├─→ index.html         (the display)                      │
│    └─→ data/last_overview.html + last_raw.json  (diagnostics)│
│                                                              │
│  Schedule */1 min → scripts/fetch_trains.py                  │
│    ├─→ urllib POST → ÖBB HAFAS mgate.exe (stdlib only)       │
│    └─→ data/trains.json  (written atomically)                │
│                                                              │
│  Web server → python3 -m http.server  (static files only)    │
└──────────────────────────────────────────────────────────────┘
            │ via Cloudflare Tunnel (optional)
            ▼
   Browser on the monitor / Smart-TV (PWA, fullscreen)
   Auto-refresh: 60 s soft · every 5 min hard (cache-bust)

The two data flows are independent. fetch_untis.py regenerates the whole index.html on each run. fetch_trains.py only writes data/trains.json, which the browser fetches every 60 s and injects into the header train widget.

Components

LayerChoiceWhy
FrontendStatic HTML5 + CSS3 + vanilla JSNo build step, loads instantly, runs anywhere.
BackendPython 3.9+ (stdlib only)No external dependencies; trivial deployment.
Data sourcesWebUntis JSON-RPC + ÖBB HAFAS mgate.exeBoth called directly via urllib, no wrapper library.
Web serverpython3 -m http.server (systemd service)Minimal footprint, enough for static files.
HostingProxmox LXC + Cloudflare TunnelInternal IP, reachable over HTTPS from outside.
Deploymentrsync from the developer workstationFast, idempotent, no CI required.
Why stdlib-only? The display runs unattended on a school PC / LXC that nobody maintains day-to-day. Every external dependency is a future breakage (pin drift, security updates, broken installs). The HAFAS train integration was deliberately written against the raw mgate.exe JSON endpoint instead of pyhafas because that library had no usable ÖBB profile.

Setup & deployment

Local run

cp config.env.example config.env   # then fill in your values
python3 scripts/fetch_untis.py     # generates index.html
python3 -m http.server 8080        # static web server
# open http://localhost:8080/
Windows / WSL note. If running under WSL without a native Python, use the Windows interpreter and force UTF-8 output: PYTHONIOENCODING=utf-8 python.exe scripts/fetch_untis.py. Timezone handling needs pip install tzdata on Windows.

Production (Proxmox LXC)

Deploy with rsync, run the web server under systemd, and schedule the fetch scripts with cron.

rsync -avz --exclude='.claude' --exclude='.git' --exclude='data/' \
      --exclude='config.env' \
      ./ root@<lxc-ip>:/var/www/supplierplan/
Security-critical exclusion. config.env is deliberately not deployed into the web root. http.server would serve every file in the directory — including the WebUntis password and Cloudflare token. Place it outside the web root instead: 
mkdir -p /etc/supplierplan
# copy config.env once to /etc/supplierplan/config.env, then:
chmod 600 /etc/supplierplan/config.env
The scripts locate it via the SUPPLIERPLAN_CONFIG environment variable (set in the crontab). Without it, they fall back to the project root — local development only.

Cron (note the env line shared by both jobs):

SUPPLIERPLAN_CONFIG=/etc/supplierplan/config.env
*/5 * * * *  cd /var/www/supplierplan && /usr/bin/python3 scripts/fetch_untis.py  >> /var/log/supplierplan-untis.log 2>&1
*   * * * *  cd /var/www/supplierplan && /usr/bin/python3 scripts/fetch_trains.py >> /var/log/supplierplan-trains.log 2>&1
Timezone caveat. If the host runs in UTC and the tzdata package is missing, the “today” calculation is wrong after ~22:00 local. On Debian LXCs run apt install tzdata.

Windows tray app

As an alternative to the LXC + cron setup, tray/ is a Windows tray application (packaged as a PyInstaller .exe) that calls the main() of both fetch scripts on timers and runs a hardened local web server. It serves only the web/ directory and is reachable on the LAN at http://<pc-ip>:<SERVER_PORT>. Cloudflare is optional.

Output is redirected into a writable data directory (next to the .exe or %LOCALAPPDATA%\Supplierplan) via SUPPLIERPLAN_CONFIG, SUPPLIERPLAN_WEBROOT and SUPPLIERPLAN_DATA — so config.env never sits inside the served web/ directory. Configuration is a tkinter window writing config.env; a green/red tray icon shows status; autostart is via an HKCU Run entry.

Build (on a developer PC with Python):

pip install pyinstaller pystray pillow
python tray/build.py
# → dist/Supplierplan/  (portable)  and, if Inno Setup is present,
#   dist/Supplierplan-Setup.exe

Multi-tenant (other schools)

The project is multi-tenant by design: a new school needs only its own config.env and a logo — no code changes.

  1. Clone and create config.env from config.env.example.
  2. Enter WebUntis access (ideally a dedicated read-only service account).
  3. Branding: SCHOOL_NAME, SCHOOL_TYPE, SCHOOL_LOCATION, PLAN_TITLE; drop in a logo and set LOGO_FILE + SHOW_LOGO=true.
  4. Region: adjust TIMEZONE. The train widget is ÖBB-specific — outside Austria set TRAIN_DISABLED=true.
  5. WebUntis quirks: list local pseudo teachers in SKIP_TEACHERS and remark codes in TEXT_BADGES; set UNTIS_DEPARTMENT_ID if departments are used.
  6. Schedule cron — index.html and manifest.json are generated from the config on every run.

Technical internals

WebUntis API

Server https://<school>.webuntis.com, JSON-RPC. Methods in active use:

MethodPurpose
authenticate / logoutCookie-based session login/logout.
getTeachersTeacher lookup (short code → first/last name).
getHolidaysHolidays/breaks for the “next school day” logic.
getTimegridUnitsPeriod grid (clock times of periods 0–10).
getLatestImportTimeUntis' own timestamp of the last change.
getSubstitutions(start, end, departmentId)The main data: the substitution plan.
weekly/data (REST)Per-teacher day detail for full-vs-partial absence — see below.

Processed fields from getSubstitutions

  • type → subst, cancel, roomchange, free
  • startTime/endTime → period number via the timegrid
  • kl[].name → class(es), deduplicated
  • te[].name/.orgid/.orgname → teacher + substitute (original struck through)
  • su[].name → subject (shown even without a class, e.g. TO (SIB))
  • ro[].name/.orgid/.orgname → room + room change (original struck through)
  • txt → remark text; badges only for the codes in TEXT_BADGES
  • lstype == "bs" → break supervision (rendered distinctly)
The orgid signal. Across teacher and room entries, orgid is the key marker that “something was replaced.” Its presence drives almost every decision below: a substitute, a room change, or an inferred absence.

Substitution processing

process_substitutions() turns the raw API list into display rows. Two filter stages ensure only real changes are shown.

Filter A — substitution level (_is_meaningful_subst)

A period is processed only if at least one of these holds:

  • type is cancel or free, or
  • some te[] entry has an orgid (someone is absent / substituting), or
  • some ro[] entry has an orgid (room change).

→ Regular lessons without a substitution (e.g. co-teacher mentions in break supervisions with no orgid at all) are skipped entirely.

Filter B — teacher level

When an explicit substitute (orgid present, name not in SKIP_NAMES) is in te[], other teachers without an orgid are dropped as co-teachers. → Eliminates team-teaching duplicates where the substitute appears twice (with and without orgid).

Heuristic: --- / Z Entfall as a predecessor marker

Pattern te=[---/SaF, BuL] means SaF is absent and BuL is present. For a teacher with no orgid, the orgname from the --- entry is adopted as the original code → display SaF → BuL.

Heuristic: FDKM-style cancellation without a substitute

For every teacher inferred absent via a --- / Z Entfall marker who is not already really covered, a virtual row is emitted: short code = the absent teacher, art = cancel, kuerzel_absent=True (render: struck-through name only, no arrow).

Why this is subtle. It also fires in mixed entries, where a real substitute covers a different teacher in the same period (e.g. supervision te=[LaI, SeM←FaM, Z Entfall←BrH]: SeM covers FaM, BrH is cancelled). An earlier not real_teacher_names guard wrongly suppressed these cancel rows, so supervisions of absent teachers vanished on project days. The fix tracks covered_orgs (every really-covered orgname) to avoid duplicate/false cancellations for genuinely covered teachers, without suppressing the legitimate ones.

Supervision type

A break supervision (lstype="bs") keeps cancel/free as its art (→ cancelled section, teacher counts as absent); only a supervision that actually takes place with a replacement becomes pause.

Separator: arrow vs. dash

  • → for a real substitution (orgid present)
  • - for txt = "Vtr. ohne Lehrer" (no real substitute; the teacher is only the class's form teacher)

Absence bars (full vs. partial)

Each section shows an “absent teachers” bar at the top. Sources: te[].orgname with orgid (the covered teacher is absent), kuerzel_absent=True (FDKM-style), and art="cancel" (cancellation ⇒ teacher absent).

Whether a teacher is shown as fully or partially absent is data-driven, not heuristic:

  • Fully absent → short code only (no period range). School convention.
  • Partially absent → range min–max of the missing periods (e.g. WöR (5–9)); a single period → just the number (SaI (6)).

determine_full_absent() calls get_teacher_present_periods(teacher_id, date) per absent teacher, hitting the REST endpoint weekly/data?elementType=2&elementId=<id>. If the teacher has no period that day with cellState ∈ {STANDARD, BREAKSUPERVISION}, they are fully absent → short code only. Otherwise → partially absent → range.

Why a separate call is needed. The substitution plan only contains periods with a change, not the whole day — so it cannot tell whether 1–5 is the full day or just the morning. Hence the extra weekly/data call.
Two field gotchas. Use cellState, not state — the element-level state reads REGULAR even for cancelled periods and is unusable. And elementId=0 is a phantom (returns arbitrary days) — always pass the real teacher ID. (get_weekly_class_absences still uses elementId=0 for classes and is therefore unreliable — a known TODO.)

Cost: one REST call per absent teacher per displayed day. On API error or unknown ID, the teacher safely falls back to the range display.

Render pipeline

Data-driven columns

Table columns are defined once in a single list COLUMNS — tuples of (key, header, css_class, cell_fn). COLGROUP, THEAD and every data row (render_row) are generated from it, so reordering the table is a one-place change. The section header's colspan uses NCOLS = len(COLUMNS). Current order: (stripe) · period · subject · class · teacher · art · room · text. Column widths stay in CSS (class-based), independent of order.

Teacher grouping & the cancelled section

Rows are grouped by the current teacher short code, one header per group (<CODE> First Last). All art=cancel rows are pulled out of the teacher groups into a single “Entfallende Stunden” (cancelled lessons) section at the end of the day.

Why split out cancellations. A teacher who only had cancellations (e.g. NeS for FDKM) would otherwise get a whole group headline for nothing. In the cancelled section the information stays compact, with the (struck-through) short code in the teacher column.

Multi-column layout (browser-side)

The server emits the list as a flat table: one <tbody data-block="teacher"> per teacher and one <tbody data-block="cancel"> per cancellation row. A JavaScript layout engine measures the available space after DOMContentLoaded and distributes the blocks across 1–4 columns with a first-fit-min strategy, setting .cols-N on the .layout-wrapper. From .cols-3, CSS switches art badges to round single letters (V/E/R/F/P).

The cancelled rows flow after all teacher blocks into the greedy fill (distributeGreedy) and break at the real column boundaries. The engine generates the section header itself (makeCancelHeader), once per column: the first affected column gets “Entfallende Stunden”, further columns get “… (Forts.)”. Each column with cancellations reserves CANCEL_HEADER_H px for the later-inserted header, so even very many cancellations (whole absent classes on project days) spread cleanly instead of overflowing one column.

Re-layout runs on resize (250 ms debounced) and on every reload. Reading order is top-left → bottom-left → top-right → bottom-right. The column cap is MAX_COLUMNS (1–4); compact mode triggers when the actual column width is below COMPACT_COL_WIDTH_PX.

The layout math has a testable Python mirror in scripts/_layout_logic.py (fit_scale, distribute_uncapped, paginate_columns), kept in sync with the JS engine.

Overflow strategy

When the plan is too full for the screen, a staged strategy kicks in — each stage individually switchable, “tomorrow” stays visible at every stage:

  1. Scale (OVERFLOW_SCALE, min factor OVERFLOW_SCALE_MIN ≈ 0.65): shrink the whole board.
  2. Reduce (OVERFLOW_REDUCE): drop the text column, then compact the cancelled section.
  3. Paginate (OVERFLOW_PAGINATE, OVERFLOW_PAGE_SECONDS ≈ 12): page through the content.

Logic lives in the JS applyLayout/layoutAll, with the testable mirror in _layout_logic.py, parsing in parse_overflow_config, and config injected as window.OVERFLOW.

Train widget (separate data flow)

fetch_trains.py POSTs to https://fahrplan.oebb.at/bin/mgate.exe — the direct JSON API the ÖBB app itself uses — and writes data/trains.json atomically (the old file survives on error). The browser fetches it every 60 s and fills #train-widget in the header.

Controlled by TRAIN_* variables. If TRAIN_STATION is empty or TRAIN_DISABLED=true, the widget is not built into the HTML at all. TRAIN_DIR_TOWARDS classifies each departure as “towards the city” vs. “away”; TRAIN_PER_DIRECTION caps the count; TRAIN_PRODUCTS filters by line prefix (e.g. only S-Bahn).

Refresh & caching

  • <meta http-equiv="refresh" content="300"> as a JS-disabled fallback.
  • JS: clock tick every second; location.reload() every 60 s (soft); every 5th tick a hard reload with ?cb=<timestamp> cache-bust (CSS/images re-fetched).
  • HTTP header Cache-Control: no-cache, no-store, must-revalidate.
  • Optional Cloudflare cache purge after each successful index.html write — purges only CLOUDFLARE_HOST if set, otherwise the whole zone.

Security

  • XSS: every value from the WebUntis API is escaped with esc() (= html.escape) before entering HTML — including fallbacks (“—”) and new fields. Each cell function escapes its own values.
  • Secrets: config.env holds the WebUntis password and Cloudflare token. It is gitignored, never deployed into the web root, and never read or quoted by the assistant. New variables are documented only in config.env.example.
  • HTTP: all Untis requests use timeout=15; the login cookie is invalidated via guaranteed logout.
